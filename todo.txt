## 1. High-confidence DELETE candidates (or stop creating)

These add complexity without long-term value.

### ❌ `/mnt/storage/photos`

**Why:**
You already have:

* `syncthing/photos/...`
* `media/...` (if Jellyfin ever indexes photos)

This standalone `photos` dir has no unique role.

**Action:**
Delete **unless** you explicitly mount it into a photo app.
If needed → symlink it to `syncthing/photos`.

```
rm -rf /mnt/storage/photos
```

---

### ❌ `/mnt/storage/duckdb`

**Why:**
DuckDB is an **embedded analytics artifact**, not primary storage.
Leaving it on HDD encourages sloppy state.

**Action:**
Move to:

```
/mnt/ssd/docker-data/duckdb
```

or kill entirely if it’s transient.

---

### ❌ `/mnt/cachehdd/jellyfin-cache`

**Why:**
Jellyfin regenerates this safely. Keeping it is fine, but naming is inconsistent.

**Action:**
Merge under a unified cache namespace (see below).

## 2. Strong MERGE / INTEGRATION opportunities

This is where you gain clarity and operational calm.

---

### 🔁 1. Download flow is fragmented

Current:

```
downloads/torrent
downloads/aria2
cachehdd/*-incomplete
```

**Better (classic layout):**

```
downloads
├── incoming      (cache, incomplete)
├── completed
│   ├── torrent
│   └── aria2
```

**Action:**

* Keep incomplete on cache HDD
* Bind-mount to `/downloads/incoming`

This reduces *mental branching* when debugging broken pipelines.

---

### 🔁 2. Media vs Syncthing duplication (BIG one)

You currently have:

* `syncthing/music`
* `media/music`
* `syncthing/audiobooks`
* `media/audiobooks`
* same for podcasts, books

This is **data schizophrenia**.

**Truth:**
Media libraries should be **authoritative** in ONE place.

**Recommended rule (old-school, proven):**

* `syncthing` = **input / personal**
* `media` = **curated / served**

**Action:**

* One-way move from `syncthing/*` → `media/*`
* Or bind-mount:

```
/mnt/storage/media/music  -> syncthing/music/albums
```

Do **not** keep both writable.

---

### 🔁 3. Cache namespace hygiene

Current cache root is a junk drawer.

**Integrate into:**

```
/mnt/cachehdd
├── downloads
│   ├── torrent
│   ├── aria2
│   └── slskd
├── media
│   ├── jellyfin
│   └── audiobookshelf
├── observability
│   ├── loki
│   ├── prometheus
│   ├── thanos
│   └── alertmanager
├── sync
│   └── syncthing-versions
└── system
    └── swapfile
```

This makes failures *legible at a glance*.

---

## 3. SSD side: minor cleanup only

Mostly good. Two tweaks.

### ⚠️ `/mnt/ssd/docker-data/cron`

**Why:**
This is infrastructure, not app data.

**Better:**

```
/mnt/ssd/system/cron
```

Keeps docker-data pure.

### ⚠️ `/mnt/ssd/docker-data/parseable`

If you keep Loki + Thanos:

* Parseable is redundant **unless** you need SQL on logs.

**Decision point:**

* Metrics stack → Prom/Loki/Thanos
* Logs analytics → Parseable
  Don’t half-commit.

These are correct and conservative.

✔ `kopia/repository` on HDD
✔ `kopia-cache` on cache disk
✔ `databases on SSD`
